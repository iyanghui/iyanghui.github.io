关于浏览器缓存

发表于 2018-04-09 | 分类于 缓存 | 阅读次数

基本认识

  • 浏览器缓存，也叫客户端缓存，具体又可以分为强缓存和协商缓存。<br/>
  • 浏览器的缓存规则，对于浏览器缓存来说，这个规则是在HTTP协议头和HTML页面的Meta标签中定义的。他们分别从新鲜度和校验值两个维度来规定浏览器是否可以直接使用缓存中的副本，还是需要去源服务器获取更新的版本。
  • 浏览器的缓存机制，其实主要就是HTTP协议定义的缓存机制（如： Expires； Cache-control等）。但是也有非HTTP协议定义的缓存机制，如使用HTML Meta 标签，Web开发者可以在HTML页面的 <head> 节点中加入 <meta> 标签，代码如下：

强缓存

  强缓存是利用Expires或者Cache-Control这两个 http response header 来实现的(启用其中一个即可，当两个同时存在的时候，Cache-Control的优先级高)，都用来表示资源在客户端的缓存有效期。强缓存并不会发请求到服务器，当浏览器对某个静态资源命中强缓存时，返回的 http 状态码为200.

1. Expires

  Expires是 http1.0 提出的一个表是资源过期的 header ，它描述的是一个绝对时间，用GMT格式的字符串表示Expires: Thu, 31 Dec 2037 23:55:55 GMT。

  • 浏览器在第一次请求静态资源的时候，服务器返回并设置响应头。浏览器在接受到后，将这个 header 一并缓存下来(也就是说，当强缓存命中的时候，response 返回的 header 其实是来自于之前浏览器缓存的 header )。

  • 再次请求这个资源的时候，先从缓存中找，比较当前时间与 Expires ,如果未过期，直接取缓存

2. Cache-Control

  Cache-Control是 http1.1 提出的新的 header ，这是一个相对时间，以秒为单位Cache-Control: max-age=31560000

  • 浏览器在第一次请求静态资源的时候，服务器返回并设置响应头。浏览器在接受到后，将这个 header 一并缓存下来(也就是说，当强缓存命中的时候，response 返回的 header 其实是来自于之前浏览器缓存的 header )。
  • 再次请求这个资源的时候，先从缓存中找，根据它第一次的请求时间和Cache-Control设定的有效期，计算出一个资源过期时间，再拿这个过期时间跟当前的请求时间比较，如果请求时间在过期时间之前，就能命中缓存。

协商缓存

  协商缓存是利用[Last-Modified, if-Modified-Since]和[Etag, if-None-Match]这两对 header 来管理的。如果 Last-Modified 和 ETag 同时被使用，则要求它们的验证都必须通过才会返回状态码304，若其中某个验证没通过，则服务器会按常规返回资源实体及状态码200。

1. Last-Modified 与 If-Modified-Since

  这一对首部通过资源修改日期判断是否有变化。服务器将资源传递给客户端时，会将资源最后更改的时间以 “Last-Modified: GMT” 的形式加在实体首部上一起返回给客户端。客户端会为资源标记上该信息，下次再次请求时，会把该信息附带在请求报文中一并带给服务器去做检查，若传递的时间值与服务器上该资源最终修改时间是一致的，则说明该资源没有被修改过，直接返回304状态码即可

  • 首次获取资源：服务器响应中添加首部Last-Modified，说明资源最近一次修改时间。浏览器保存下来这个时间
  • 验证资源：浏览器带上If-Modified-Since首部，值就是Last-Modified返回的时间；服务器比较If-Modified-Since的时间与资源最近一次修改时间;
  • 如果最近一次修改时间小于If-Modified-Since的时间，则返回304响应，响应中也会有cache-control、expires、Date首部；
  • 如果最近一次修改时间大于If-Modified-Since的时间，则返回新资源，并且更新所对应的全部首部，包括cache-control、expires、Date

2. ETag

  ETag是服务器为资源生成的唯一标识字符串(这个唯一标识是一个字符串，只要资源有变化这个串就不同，跟最后修改时间没有关系)。使用过程与Last-Modified类似。

  • 浏览器第一次跟服务器请求一个资源，服务器在返回这个资源的同时，在respone的header加上ETag的header
  • 浏览器再次跟服务器请求这个资源时，在request的header上加上If-None-Match的header，这个header的值就是上一次请求时返回的ETag的值
  • 服务器根据资源生成一个新的ETag，和传过来的If-None-Match进行比较，如果这两个值相同就说明资源没有变化，返回304 Not Modified，但是不会返回资源内容；如果有变化，就正常返回资源内容。与Last-Modified不一样的是，当服务器返回304 Not Modified的响应时，由于ETag重新生成过，response header中还会把这个ETag返回，即使这个ETag跟之前的没有变化：
